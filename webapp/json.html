Motivation

 The RFC 7089 defines a TimeMap for an Original Resource as a resource from which a list of URIs of Mementos of the Original Resource is available. It specifies the content that needs to be expressed in a TimeMap and defines a serialization based on the application/link-format media type, which itself is based on the definition of content for the HTTP Link header field. This TimeMap format was chosen because Memento clients must be able to parse HTTP Link headers anyhow when engaging in datetime negotiation. However, the format is not particularly appealing for consumption by, for example, JavaScript applications. In order to meet the needs of such applications, a JSON TimeMap format is defined in this document.

TimeMap - Basic Format
This section details how a basic TimeMap should be serialized as a JSON document with MIME type application/json. The object must contain the URI of the Original Resource (original_uri) and it must list the Mementos for that resource. All Mementos known by the server that provides the TimeMap must be listed in an array (mementos.all). For each Memento, its URI (uri) and its archival/version datetime (datetime) must be provided.

Some key-values are not mandatory, yet highly recommended. The TimeMap should also contain the URI of the TimeGate for the Original Resource (timegate_uri) and the URI of the TimeMap itself (timemap_uri), serialized as application/json (timemap_uri.json_format) and/or application/link-format (timemap_uri.link_format). In addition, the first and last Mementos known by the responding server should be provided separately (mementos.first, and mementos.last, respectively).

Example 1: Assuming http://example.com/ is an Original Resource for which http://an.archive.org/timemap/json/http://example.com/ is a JSON formatted TimeMap. A TimeMap serialization that contains all required and recommended information looks as follows: 
{
    "original_uri": "http://example.com/",
    "timegate_uri": "http://an.archive.org/timegate/http://example.com/",
    "timemap_uri": {
        "json_format": "http://an.archive.org/timemap/json/http://example.com/",
        "link_format": "http://an.archive.org/timemap/link/http://example.com/"
    },
    "mementos": {
        "first": {
            "datetime": "2000-06-21T04:41:56Z",
            "uri": "http://an.archive.org/20000621044156/http://example.com/"
        },
        "last": {
            "datetime": "2012-08-31T12:22:34Z",
            "uri": "http://an.archive.org/20120831122234/http://example.com/"
        },
        "all": [
            {
                "datetime": "2000-06-21T04:41:56Z",
                "uri": "http://an.archive.org/20000621044156/http://example.com/"
            },
            {
                "datetime": "2008-04-10T20:30:51Z",
                "uri": "http://an.archive.org/20080410203051/http://example.com/"
            },
            {
                "datetime": "2012-08-31T12:22:34Z",
                "uri": "http://an.archive.org/20120831122234/http://example.com/"
            }
        ]
    }
}

API Format

 Web archive APIs allows the user to request a Memento based on two things: Original Resource URI and a date time. This section details how an API response for such request should be serialized. The response must contain the URI to the Original Resource (original_uri) and its closest (nearest in time to the requested date) Memento that the server knows about (mementos.closest). For each Memento, its URI (uri) and its archival/version datetime (datetime) must be provided.

It is recommended that the reponse also contains the URI of the TimeGate for the Original Resource (timegate_uri) and the URI of the TimeMap (timemap_uri), serialized as application/json (timemap_uri.json_format) and/or application/link-format (timemap_uri.link_format). In addition, the first and last Mementos known by the responding server should be provided separately (mementos.first, and mementos.last, respectively).

Example 2: Assume that a user requests a Memento for http://www.example.com/ at date time 2008-05-09T00:00:00Z against an archive API. The API responds with the Memento that is the closest in time to the requested date that it knows about (which in our example was captured at 2008-04-10T20:30:51Z). 
{
    "original_uri": "http://example.com/",
    "timegate_uri": "http://an.archive.org/timegate/http://example.com/",
    "timemap_uri": {
        "json_format": "http://an.archive.org/timemap/json/http://example.com/",
        "link_format": "http://an.archive.org/timemap/link/http://example.com/"
    },
    "mementos": {
        "first": {
            "datetime": "2000-06-21T04:41:56Z",
            "uri": "http://an.archive.org/20010621044156/http://example.com/"
        },
        "prev": {
            "datetime": "2005-04-10T07:05:12Z",
            "uri": "http://an.archive.org/20050410070512/http://example.com/"
        },
        "closest": {
            "datetime": "2008-04-10T20:30:51Z",
            "uri": "http://an.archive.org/20080410203051/http://example.com/"
        },
        "next": {
            "datetime": "2009-04-10T23:45:20Z",
            "uri": "http://an.archive.org/20090410234520/http://example.com/"
        },
        "last": {
            "datetime": "2012-08-31T12:22:34Z",
            "uri": "http://an.archive.org/20120831122234/http://example.com/"
        }
    }
}

TimeMap Paged Format

 In cases where many Mementos for an Original Resource exist on a server, its TimeMap can become large and it is then preferred to split it into pages. A paged TimeMap must contain the URI of the Original Resource (original_uri). It must also provide the list of all Mementos that are in this page (mementos.all) and references to the next and the previous pages if they apply (pages.next, pages.prev, respectively). Any page Object must contain the URI (uri) to the page, and the timespan that it covers (from and until) which are the datetimes of its first and last Memento respectively.

Here again, it is recommended that the reponse also contains the URI of the TimeGate for the Original Resource (timegate_uri) and the URI of the TimeMap itself (timemap_uri), serialized as application/json (timemap_uri.json_format) and/or application/link-format (timemap_uri.link_format). In this case, first and last Mementos are the ones in contained in this page. They should be provided separately (mementos.first, and mementos.last).

Example 3: Assume a TimeMap for the Original Resource http://example.com/ is split into 3 pages which can be accessed at the server http://an.archive.org/timemap/X/, where X is the page number. This example shows how the TimeMap #2 should look. This page covers the time from 2003-05-13T00:00:13Z to 2006-10-05T20:30:51Z (archival/versions datetimes for its first and last Mememtos). In this case, all older Mementos availible are contained in the pages.prev and all newer are contained in pages.next. Note that the values for pages.prev and pages.next can be empty for the first and last page respectively. 
{
    "original_uri": "http://example.com/",
    "timegate_uri": "http://an.archive.org/timegate/http://example.com/",
    "timemap_uri": {
        "json_format": "http://an.archive.org/timemap/2/json/http://example.com/",
        "link_format": "http://an.archive.org/timemap/2/link/http://example.com/"
    },
    "mementos": {
        "first": {
            "datetime": "2003-05-13T00:00:13Z",
            "uri": "http://an.archive.org/20030513000013/http://example.com/"
        },
        "last": {
            "datetime": "2006-10-05T20:30:51Z",
            "uri": "http://an.archive.org/20061005203051/http://example.com/"
        },
        "all": [
            {
                "datetime": "2003-05-13T00:00:13Z",
                "uri": "http://an.archive.org/20030513000013/http://example.com/"
            },
            {
                "datetime": "2005-04-10T07:05:12Z",
                "uri": "http://an.archive.org/20050410070512/http://example.com/"
            },
            {
                "datetime": "2006-10-05T20:30:51Z",
                "uri": "http://an.archive.org/20061005203051/http://example.com/"
            }
        ]
    },
    "pages": {
        "prev": {
            "from": "2000-06-21T04:41:56Z",
            "until": "2003-05-12T22:25:15Z",
            "uri": "http://an.archive.org/timemap/1/json/http://example.com/"
        },
        "next": {
            "from": "2006-10-06T00:00:57Z",
            "until": "2009-04-10T23:45:20Z",
            "uri": "http://an.archive.org/timemap/3/json/http://example.com/"
        }
    }
}

TimeMap Index Format

 For the same reasons as in a TimeMap paging, TimeMap indexing can be used to handle high volume TimeMaps. The main difference between a page and an index is that the later does not contain any Memento itself, but only references to other TimeMaps (typically 2 or more). As always, an index TimeMap must contain the URI of the Original Resource original_uri. The referrenced TimeMaps must be placed in an array (indexes) and each of such Object must express the timespan that it covers (from and until), plus its URI (uri).

It is also recommended to provide the URI of a TimeGate for the Original Resource (timegate_uri), and the the URI of the root TimeMap (timemap_uri).

Example 4: Assume that three TimeMaps describe which Mementos are availible for the Original Resource (http://example.com/). Each TimeMap can be accessed at http://anX.archive.org/timemap/json/http://example.com/ where X is the TimeMap number. All three are referred to in a root TimeMap, which can be accessed at http://an.archive.org/timemap/link/http://example.com/. This example shows how such index should look. 
{
    "original_uri": "http://example.com/",
    "timegate_uri": "http://an.archive.org/timegate/http://example.com/",
    "timemap_uri": {
        "json_format": "http://an.archive.org/timemap/json/http://example.com/",
        "link_format": "http://an.archive.org/timemap/link/http://example.com/"
    },
    "indexes": [
        {
            "from": "2000-06-21T04:41:56Z",
            "until": "2003-05-12T22:25:15Z",
            "uri": "http://an1.archive.org/timemap/json/http://example.com/"
        },
        {
            "from": "2003-05-13T00:00:13Z",
            "until": "2006-10-05T20:30:51Z",
            "uri": "http://an2.archive.org/timemap/json/http://example.com/"
        },
        {
            "from": "2006-10-06T00:00:57Z",
            "until": "2012-08-31T12:22:34Z",
            "uri": "http://an3.archive.org/timemap/json/http://example.com/"
        }
    ]
}
